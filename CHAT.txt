Great research material. Here's a clear breakdown of what I found:

Gmail API Permission Scopes
There are essentially three scopes relevant to your use case, each with different power levels:

gmail.labels (non-sensitive) This is specifically for managing label definitions — creating, renaming, deleting, or modifying the labels themselves. It does not grant the ability to apply or remove labels from individual messages. So despite the name, it won't do what you want.

gmail.modify (restricted) This is the scope you'd need. It allows reading messages, composing/sending, and importantly — adding and removing labels on individual messages. Google explicitly documents that this scope "does not allow immediate, permanent deletion," which is the key distinction. It's the right scope for your goal.

mail.google.com/ (restricted, full access) This is the nuclear option — it includes everything in gmail.modify plus the ability to permanently and immediately delete messages, bypassing the trash entirely. You'd want to avoid this one.

Your Specific Concern: Trash and 30-Day Deletion
Your instinct was correct to worry about this. Here's how it works:

Gmail has a system label called TRASH. When gmail.modify is granted, the API does expose a messages.trash endpoint that moves messages to trash (effectively adding the TRASH label), which starts Gmail's automatic 30-day deletion clock. So gmail.modify does technically enable soft-deletion via trash, even though it can't permanently delete anything.

The good news: trash is fully reversible within 30 days via messages.untrash. But if you want zero risk of even that, the mitigation is straightforward — in your connector implementation, simply don't expose the trash endpoint and explicitly block adding the TRASH or SPAM system labels. The scope itself doesn't prevent this, but you can enforce it at the connector layer.

Recommendation for Your Connector
Use gmail.modify as the OAuth scope, then in the connector's implementation:

Expose messages.modify for adding/removing labels (this is the endpoint the existing connector's gmail_list_labels tool even references for future use)
Explicitly filter out TRASH and SPAM from any labels that can be applied
Don't expose messages.trash, messages.delete, or threads.delete endpoints
This gives you full label add/remove capability with no path to permanent deletion and no path to even accidental soft-deletion via trash.

Good questions — here's the full picture across all three:

What format is a connector?

It's essentially a small web server you write and host yourself. You'd build it in either TypeScript or Python using Anthropic's MCP SDK, and it needs to expose an HTTP endpoint (supporting either SSE or Streamable HTTP transport). Cloudflare Workers is a very popular hosting choice because it handles autoscaling, has a generous free tier, and has built-in OAuth token management support. But you could also host it on any server/cloud provider. There's no special packaging format — it's just a running web service at a URL that speaks the MCP protocol.

Is there source code for the official Gmail connector to fork?

No — the official Anthropic Gmail connector appears to be a proprietary hosted service, not open source. However, there are community-built Gmail MCP servers on GitHub you could use as a starting point. The most notable one is cafferychen777/gmail-mcp, though it takes an unusual approach (piggybacking on your Chrome session rather than using the Gmail API directly, which means no OAuth setup needed but also means it requires a browser to be running). For your use case — a proper API-based connector — it would be a useful reference for structure but not a direct fork.

Do you need to register with Google for OAuth?

Yes, you do need to create a Google Cloud project and register an OAuth app. The good news is how much this matters depends heavily on your intended audience:

For personal use only (just you as the user): You can keep the app in "Testing" mode indefinitely, add yourself as a test user, and never go through any verification process at all. gmail.modify works fine this way. This is realistically the path for a personal productivity connector.
For a small group of known users (friends, a small team): Google makes allowances for this as well — the key phrase in their docs is "if your app is used by only a few users, all of whom are known personally to you." You can stay in Testing mode, though there's a cap on test users (100).
For public/directory submission (anyone can connect): This is where it gets significant. gmail.modify is a "restricted" scope, meaning Google requires a full security assessment through their App Defense Alliance framework, brand verification, annual reassessment — the whole enterprise compliance process. This is weeks of work and real cost.
Bottom line for your situation: If you're building this for yourself (and potentially submitting it to the Anthropic directory for others to use with their own Google OAuth credentials), the architecture works like this — each user who installs your connector would authenticate with their own Google account, creating their own OAuth tokens. You'd just need your own Google Cloud project and app registration to obtain a client ID/secret, and it would stay in testing mode for personal use without needing Google's full verification. The heavy verification only kicks in if you want Google to show users a clean consent screen without the "unverified app" warning.

The cost picture here is genuinely murky, with a wide range depending on the tier of assessment required. Here's the full breakdown:

The Process (Steps & Work Required)
Step 1 — Google Cloud Project Setup (a day or two of work) Create a project in Google Cloud Console, enable the Gmail API, configure an OAuth consent screen, add your requested scopes, set up brand verification (requires a domain you own with specific DNS or file verification). Typically 2–3 business days for Google to approve brand verification.

Step 2 — Build & Document Your App (weeks of development work) You need a working, demo-able app before you can even submit for verification. Google requires you to submit an unlisted YouTube video actually demonstrating the OAuth flow and showing every restricted scope being used in context. You also write documentation justifying each scope.

Step 3 — Submit for Restricted Scope Review (2–6 weeks of waiting + back-and-forth) Google reviews your submission. They frequently come back with questions or require changes. This step alone can take several weeks.

Step 4 — CASA Security Assessment (2–8 weeks, the expensive part) Because gmail.modify is a restricted scope, Google requires a third-party security audit through their App Defense Alliance (authorized labs include TAC Security, Leviathan, DEKRA, Prescient Security, and others). This is where the cost lives.

Step 5 — Annual Recertification Every 12 months, you repeat the CASA assessment to maintain verified status. It's an ongoing cost, not a one-time thing.

The Costs
This is where the picture gets murky — there's a wide range in what's reported:

CASA has three tiers, and which tier you're assigned depends on how many users your app has and how sensitive the data access is:

Tier 2 (mid-level, authorized scanners): roughly $540–$1,800/year per assessors like TAC Security
Tier 3 (full lab audit, required for marketplace listings): roughly $4,500/year
However, Nylas's writeup on the process (they went through it themselves) quotes Google's own documentation as citing $15,000–$75,000 for security assessments, and notes Nylas was able to do theirs in two weeks partly because they already had SOC 2 Type II certification. The gap between the ~$1,500 tier 2 quotes and the $15k+ quotes is genuinely confusing — it likely reflects that enterprise-scale apps with many users face much stricter requirements, while smaller apps may qualify for lower tiers.

The honest summary of out-of-pocket cost for a publicly-listed connector: probably somewhere between $1,000 and $5,000 per year at the low end if you qualify for Tier 2, potentially significantly more if Google assigns you to a higher tier. Plus your time — realistically 4–12 weeks of elapsed time from start to finish, with active work spread throughout.

The Practical Takeaway for Your Situation
For a connector you'd submit to the Anthropic directory — where users authenticate with their own Google credentials — the good news is that the heavy compliance burden may not apply to you the same way. Google's verification is about your app accessing user data at scale. If users are essentially self-hosting the OAuth credentials (using their own Google Cloud project), or if you keep the app in Testing mode while the Anthropic connector directory handles distribution, the compliance calculus changes.

It might be worth looking at how the existing Anthropic Gmail connector handles its Google OAuth registration — whether Anthropic is running one centralized verified Google app or whether each user provides their own credentials. That answer would determine whether you'd need to go through CASA at all, or whether you could piggyback on a similar model.

What to Include in a Public Organization Connector
Definitely yes:

messages.modify and messages.batchModify — the whole point
threads.modify — this is arguably more important than single-message modify, since Gmail's UI is thread-based and users think in terms of conversations
labels.list / labels.create — if Claude is helping organize email intelligently, being able to create a new label on the fly ("Project: Kitchen Renovation") is a big value-add
history.list — this one's underrated but crucial for a good connector. Instead of re-scanning the whole inbox every time, you can ask "what changed since last time?" making it far more efficient and enabling reliable background organization
Maybe, with caveats:

labels.update / labels.patch — useful for renaming/recoloring labels, but gives Claude the ability to rename your existing labels, which could be disorienting if done carelessly
labels.delete — deleting a label definition removes it from all messages simultaneously. Powerful but risky. Probably exclude, or require extra confirmation
drafts.create — drafting a suggested reply as part of organizing a thread could be nice, but it's scope creep for a pure organization tool
Definitely exclude for a public connector:

messages.send / drafts.send — sending email on someone's behalf is extremely sensitive. If your connector gets compromised or misbehaves, it could send spam or phishing mail from the user's account. This is the single most dangerous capability in the list and has no place in an organization tool
messages.trash / threads.trash — as discussed, even soft-deletion is something to block at the implementation layer
messages.import / messages.insert — these are for migrating mail from other providers or injecting synthetic messages; no legitimate use in an organizer
labels.delete — too destructive to expose broadly
The send exclusion in particular would be significant for Google's OAuth review. Apps that request gmail.modify but explicitly don't use the send or delete endpoints are in a much better position during verification, since the main thing Google worries about with restricted scopes is spam/phishing/data exfiltration.

One real gap: messages.attachments.get

This one is genuinely missing and matters.
